# 🧩 Jieum — 신뢰 기반 스터디 매칭 플랫폼

> 백석대학교 학생들을 위한 스터디 매칭 서비스 | 2024 SmaIt 경진대회 입상작

<br>

## 🔍 왜 만들었나요?

대학교 커뮤니티 앱 **에브리타임**에서 스터디원을 구하는 게시글을 본 적 있으신가요?

올려도 반응이 없거나, 연락이 왔어도 실제 참여로 이어지지 않는 경우가 많습니다.  
익명 기반 커뮤니티 특성상, **처음 보는 사람을 믿고 함께 공부한다는 것** 자체에 심리적 장벽이 존재합니다.

저희 팀은 이 문제를 직접 경험했고, 이렇게 질문했습니다.

> _"신뢰할 수 있는 스터디 환경을 만들 수 있다면?"_

**Jieum**은 그 질문에서 출발했습니다.  
단순한 모집 게시판이 아닌, **사용자 정보와 스터디 활동 이력을 기반으로 신뢰를 쌓아가는 매칭 플랫폼**을 목표로 설계했습니다.

<br>

## 🎯 Problem & Solution

| ❗ Problem | 💡 Solution |
|---|---|
| 익명 커뮤니티에서의 신뢰 부족 | 학교 계정 기반 실명 서비스로 신뢰도 확보 |
| 상대방의 성실도 파악 불가 | **스터디 온도** 시스템으로 활동 신뢰 지표 제공 |
| 스터디 모집·참여 경로 부재 | 스터디 모집 게시글 + 매칭 기능 제공 |

> 💡 **스터디 온도란?**  
> 스터디 완료 후 스터디장과 팀원이 서로 평가하는 신뢰 지표입니다.  
> 이후 스터디 참여 선발 시 참고 기준으로 활용됩니다.

<br>

## 🙋 내가 맡은 역할

**마이페이지** 전체 UI 및 기능 구현
- 사용자 정보 표시 (이름, 학부, 학번, 관심 키워드)
- 열정 온도 Progress Bar
- 내 스터디 탭 관리 (진행 중 / 완료 / 전체)
- 각 단계별로 성장하는 캐릭터 디자인

**게시글 상세 페이지** 구현
- 스터디 정보 UI (세부 영역, 기간, 인원, 주차별 일정)
- 댓글 / 답글 시스템
- 좋아요(관심) 반응 기능
- 모집 중 / 모집 완료 상태 UI

<br>

## 🛠 기술 스택 & 선택 이유

| 기술 | 선택 이유 |
|---|---|
| **React** | 컴포넌트 단위로 UI를 분리해 역할별로 관리하기 위해 선택. `useState` / `useEffect`로 댓글, 좋아요 등 상태를 각 컴포넌트에서 독립적으로 처리 |
| **React-Bootstrap** | 첫 React 프로젝트인 만큼, 레이아웃 구성보다 기능 구현에 집중하기 위해 UI 컴포넌트 라이브러리 활용 |
| **MUI Icons** | 별도 아이콘 제작 없이 의미 전달력 높은 아이콘을 빠르게 적용 (프로필, 좋아요, 댓글 아이콘 등) |
| **SweetAlert2** | 기본 `alert()`은 UX 측면에서 한계가 명확했음. 신청 확인처럼 되돌릴 수 없는 액션에 커스터마이징 가능한 모달이 필요해 도입 |

> 별도의 데이터 패칭 라이브러리(Axios, React Query 등)는 사용하지 않았으며,  
> 로컬 상태(`useState`) 기반으로 기능을 구현했습니다.

## 🧠 Troubleshooting & 고민의 흔적들

### 1️⃣ 댓글과 답글, 어떻게 하면 깔끔하게 관리할 수 있을까?

처음엔 댓글 배열과 답글 배열을 **분리된 state로 관리**하려 했습니다.

```js
// 초기 설계 (문제 있는 구조)
const [comments, setComments] = useState([]);
const [replies, setReplies] = useState([]);
```

하지만 막상 구현해보니, **어떤 댓글에 달린 답글인지** 연결 고리가 명확하지 않았습니다.  
인덱스로 맞추려다 보니 렌더링 로직도 복잡해졌고, 나중에 답글 개수를 합산하는 것도 번거로웠습니다.

고민 끝에 댓글 객체 안에 `replies` 배열을 포함하는 구조로 변경했습니다.

```js
// 개선된 구조
{
  text: "댓글 내용",
  replies: [],         // 답글은 댓글에 종속
  userID: "haneul1219",
  userName: "고하늘",
  timestamp: "2024.xx.xx 14:30"
}
```

이후 총 댓글 수 계산도 자연스럽게 해결됐습니다.

```js
const totalCommentCount = comments.reduce(
  (count, comment) => count + 1 + comment.replies.length, 0
);
```

**배운 점:** 처음 데이터 구조를 어떻게 잡느냐가 이후 UI 렌더링 방식과 코드 복잡도를 결정짓는다는 것을 직접 체감했습니다. 기능을 먼저 생각하기보다 **데이터 흐름을 먼저 설계하는 습관**이 중요하다는 걸 이 경험으로 처음 깨달았습니다.

---

### 2️⃣ "공백만 입력해도 댓글이 등록된다" — 사소하지만 사소하지 않은 문제

기능 구현 후 테스트를 하다가 스페이스바만 눌러도 댓글이 등록되는 걸 발견했습니다.  
비어있는 댓글이 리스트에 쌓이는 건 사용자 입장에서 당연히 이상한 경험입니다.

```js
// 수정 전
const handleAddComment = () => {
  if (commentText) { ... }
}

// 수정 후
const handleAddComment = () => {
  if (commentText.trim()) { ... }  // 앞뒤 공백 제거 후 검증
}
```

단 한 줄의 수정이었지만, 이 작업을 하면서 **"사용자가 실수할 수 있는 모든 경우를 생각해봐야 한다"** 는 걸 배웠습니다.  
작은 디테일 하나가 서비스 완성도에 대한 인상을 크게 바꾼다는 것도요.

---

### 3️⃣ 댓글에서 게시글 작성자를 어떻게 표시할까?

댓글이 여러 개 달렸을 때, 어떤 댓글이 **게시글 작성자의 댓글**인지 구분이 안 된다는 걸 UX 관점에서 문제로 인식했습니다.

그래서 댓글 프로필 컴포넌트에 `authorID`를 prop으로 전달하고,  
현재 댓글의 `userID`와 비교해 "작성자" 뱃지를 조건부 렌더링했습니다.

```js
// CommentProfile.jsx
const isAuthor = userID === authorID;

return (
  <span className="comment-author">
    {userName}
    {isAuthor && <span className="author-badge">작성자</span>}
  </span>
);
```

**배운 점:** "기능이 된다"와 "사용자가 직관적으로 이해한다"는 다릅니다.  
보는 사람 입장에서 **어떤 정보가 필요한지**를 먼저 생각하고 UI를 설계해야 한다는 걸 배웠습니다.

---

### 4️⃣ 신청 버튼, 누르면 바로 완료? — 되돌릴 수 없는 액션엔 확인이 필요하다

초기엔 신청 버튼을 누르면 즉시 신청이 완료되었습니다.  
그런데 생각해보면, **실수로 눌렀을 때 취소할 방법이 없다는 건 UX 측면에서 큰 문제**입니다.

```js
Swal.fire({
  title: '신청 완료 하시겠습니까?',
  showCancelButton: true,
  confirmButtonColor: '#DADEAA',
  cancelButtonColor: '#FF5555',
  confirmButtonText: '신청',
  cancelButtonText: '취소',
}).then((result) => {
  if (result.isConfirmed) {
    Swal.fire({
      title: '신청 완료되었습니다',
      text: '마이페이지에서 내역을 확인해 보세요',
      icon: 'success',
    });
  }
});
```

SweetAlert2를 도입해 확인 → 완료 피드백의 2단계 흐름을 만들었습니다.  
기본 `window.confirm()`보다 시각적으로도 자연스럽고, 브랜드 색상도 적용할 수 있었습니다.

**배운 점:** 사용자의 액션에는 **명확한 피드백**이 반드시 따라야 합니다.  
"지금 무슨 일이 일어났는지"를 사용자가 인지할 수 있어야 신뢰할 수 있는 서비스가 됩니다.

---

### 5️⃣ 모집 상태, 보는 사람이 바로 알 수 있어야 한다

게시글을 봤는데 이미 모집이 끝난 스터디라면? 사용자 입장에서 굉장히 허탈합니다.  
모집 상태를 텍스트로만 표현하는 건 한계가 있다고 판단해, **버튼 비활성화 + 텍스트 변경**으로 상태를 명확히 시각화했습니다.

```js
const [isRecruiting, setIsRecruiting] = useState(true);

<Button disabled={!isRecruiting} className="toRecruit-state">
  {isRecruiting ? '모집 중' : '모집 완료'}
</Button>
```

**배운 점:** 상태(State)는 단순히 데이터를 저장하는 것이 아니라, **UI가 어떻게 보여야 하는지를 결정하는 설계 요소**라는 것을 이해하게 됐습니다.

## 📈 이 프로젝트를 통해 성장한 것들

이 프로젝트는 제가 처음으로 팀과 함께 완성한 React 웹 서비스입니다.

기술적인 성장도 있었지만, 그보다 더 크게 배운 건 **"왜 이렇게 만들어야 하는가"를 스스로 질문하는 태도** 였습니다.

기능을 단순히 동작하게 만드는 것과, 사용자가 자연스럽게 이해하고 사용할 수 있게 만드는 것은 다릅니다. 댓글 구조를 고치고, 공백 하나를 막고, 모달 하나를 추가하는 과정에서 — 작은 결정 하나하나가 사용자 경험을 어떻게 바꾸는지를 직접 느꼈습니다.

앞으로도 "이게 돌아가는가"보다 **"이게 사용자에게 좋은 경험인가"** 를 먼저 묻는 개발자가 되고 싶습니다.
